# Loan Payback Prediction

---

ML-проект по бинарной классификации посвящён прогнозированию того, **вернёт ли клиент кредит**, 
на основе его профиля и финансовых характеристик.

**Языки:** [English](README.md) | [Русский](README_RU.md)

---

## Быстрый старт

Проект представляет собой воспроизводимый пайплайн обучения модели и HTTP API для онлайн-инференса.

**Требования:**

- Python 3.8+ (протестировано на Python 3.12);
- Docker (опционально, для запуска API в контейнере)

### 1) Клонирование репозитория

```
git clone https://github.com/lid11a/loan-payback-predictor.git
cd loan-payback-predictor
```

### 2) Настройка токена Kaggle API (однократно)

- **Windows:** `%USERPROFILE%\.kaggle\kaggle.json`
- **Linux / macOS:** `~/.kaggle/kaggle.json`

For Linux/macOS, file permissions must also be set:

```
 chmod 600 ~/.kaggle/kaggle.json
```

### 3) Создание виртуального окружения и установка зависимостей

**Windows (PowerShell)**

```
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements-all.txt
```

**macOS / Linux**

```
python -m venv .venv
source .venv/bin/activate
pip install -r requirements-all.txt
```

### 4) Обучение модели

```
python -m src.models.train
```

### 5) Запуск API локально (без Docker)

```
uvicorn src.api.app:app --host 0.0.0.0 --port 8000
```

**Swagger UI** доступен по адресу:

http://127.0.0.1:8000/docs

Перед отправкой запроса посмотрите, какие признаки ожидает модель (`GET /features`),
и какие значения допустимы для категориальных признаков (`GET /features/schema`).

Пример запроса для инференса одного объекта: [Пример запроса к API](#пример-запроса-к-api).

Для остановки API нажмите `Ctrl+C`.

## Общая идея проекта

Проект посвящён задаче кредитного скоринга и демонстрирует полный цикл работы
с ML-задачей бинарной классификации: от исследовательского анализа данных
и экспериментов с моделями до воспроизводимой кодовой реализации и сервисного инференса.

Цель проекта — продемонстрировать процесс выбора модели и переход от исследовательских выводов 
к инженерному ML-решению, пригодному для практического использования и дальнейшего развития.

В рамках проекта выполнены:

- исследовательский анализ данных, включая изучение структуры признаков,
  целевой переменной и факторов, влияющих на риск невозврата кредита;
- статистический анализ и проверка гипотез, используемых для интерпретации данных
  и сопоставления статистических выводов с поведением моделей;
- экспериментальное сравнение различных классов моделей и подходов к обучению;
- выбор финального решения с учётом заданного бизнес-ограничения
  (подбор порога классификации по целевому FPR);
- перенос выбранных решений в воспроизводимый ML-пайплайн
  с единым препроцессингом и контролем консистентности обучения и инференса;
- реализация сценариев обучения и инференса с трекингом экспериментов в MLflow;
- реализация HTTP API для онлайн-предсказаний на базе FastAPI;
- настройка автоматического тестирования и CI для контроля корректности кода;
- реализация воспроизводимого запуска сервиса с использованием Docker;
- реализация offline-мониторинга дрифта признаков (PSI)
  для контроля стабильности входных данных.

Далее проект последовательно рассматривается на уровне исследовательской части, кодовой реализации 
и вспомогательных компонентов, обеспечивающих воспроизводимость и контроль качества решения.

## Исследовательская часть проекта

Исследовательская часть проекта выполнена в ноутбуке `notebooks/loan_payback_predictor.ipynb`
и является основой для всех последующих решений по построению ML-пайплайна и выбору моделей.

В ноутбуке последовательно рассмотрены основные этапы анализа данных и моделирования.

### Анализ данных (EDA)

Первичный анализ данных направлен на понимание структуры датасета, 
свойств признаков и поведения целевой переменной.

В рамках EDA выполнены:

- анализ структуры датасета, типов признаков и целевой переменной;
- исследование распределений числовых признаков и выявление выбросов;
- анализ пропусков и оценка их потенциального влияния;
- анализ категориальных признаков, включая:
  - распределения категорий;
  - уровень невозврата кредита по категориям.

### Статистический анализ

Статистический анализ использовался для количественной оценки связи признаков с целевой переменной 
и проверки выявленных на этапе EDA закономерностей.

В рамках анализа выполнены:

- проверка зависимости категориальных признаков и целевой переменной с использованием статистических тестов;
- интерпретация статистической значимости признаков;
- сопоставление статистической значимости признаков с оценками их вклада в модели.

### Исследование структуры данных

 Для более глубокого понимания структуры клиентской базы
кластеризация применялась как исследовательский инструмент.

**Применяемые методы:**

- KMeans;
- Gaussian Mixture Models (GMM).

**Анализ результатов кластеризации включал:**

- подбор числа кластеров по silhouette score;
- анализ распределения целевой переменной внутри кластеров;
- визуализацию кластерной структуры в пониженной размерности (PCA);
- анализ распределения признака `grade_subgrade` внутри кластеров.

Кластеризация использовалась исключительно в аналитических целях и не является частью финального ML-пайплайна.

### Эксперименты с моделями

Эксперименты с моделями были направлены на сравнение различных классов алгоритмов
с точки зрения качества предсказаний, устойчивости и чувствительности к признакам.

В рамках экспериментов рассмотрены следующие подходы:

- линейные модели;
- методы, основанные на расстояниях и разделяющих гиперплоскостях (KNN, SVM);
- вероятностные модели (Naive Bayes);
- деревья решений и ансамблевые методы;
- нейронная сеть (MLP).

Эксперименты проводились с различными вариантами препроцессинга и наборами признаков 
с целью оценки устойчивости моделей и выявления наиболее подходящего подхода для данной задачи.

### Выбор финального решения

Выбор финального решения осуществлялся на основе результатов
экспериментального сравнения моделей и анализа их поведения.

Для финального решения были реализованы следующие шаги:

- применение early stopping для контроля переобучения;
- анализ важности признаков и интерпретация полученных результатов;
- подбор порога классификации в соответствии с заданным бизнес-ограничением (целевой FPR).

## Кодовая реализация проекта

Кодовая реализация проекта переносит результаты исследовательского анализа
в воспроизводимый ML-пайплайн и сервисную логику для инференса.

Основной задачей кодовой реализации является обеспечение единообразия логики обработки данных 
и применения модели при обучении и инференсе, воспроизводимости экспериментов 
и возможности практического использования выбранного решения.

### Обучение и инференс

Обучение и инференс реализованы как отдельные сценарии, использующие единый пайплайн обработки данных.

В процессе обучения:

- формируется единый препроцессинг данных;
- обучается финальная модель с использованием механизмов контроля переобучения (early stopping);
- подбирается порог классификации в соответствии с выбранным бизнес-ограничением (целевой FPR);
- сохраняется единый bundle, включающий модель, препроцессор, порог классификации и вспомогательные метаданные.

Инференс выполняется с использованием сохранённого bundle, 
что исключает расхождения в логике обработки данных и применения модели.

### Воспроизводимость и логирование

В проекте используются два независимых механизма фиксации результатов работы.

Для отслеживания выполнения кода применяется стандартное логирование Python.
Логи формируются по основным модулям проекта и сохраняются в файловом виде,
что позволяет анализировать ход выполнения пайплайна, сервисной логики и диагностировать ошибки.

Для фиксации результатов обучающих запусков используется MLflow.
В рамках каждого запуска сохраняются параметры обучения, метрики качества и артефакты модели, 
что обеспечивает воспроизводимость экспериментов и возможность сравнения различных запусков.

### Сервис для онлайн-инференса

Для онлайн-инференса реализован HTTP API на базе FastAPI.

Сервис:

- загружает сохранённый bundle модели при старте;
- выполняет валидацию структуры входных запросов;
- возвращает вероятности и бинарное решение в соответствии с заданным порогом классификации.

### Тестирование и контроль качества

Ключевая логика ML-пайплайна и сервиса покрыта автоматическими тестами (pytest).

Тестами проверяются:

- загрузка и чтение данных (`test_download.py`, `test_load.py`);
- препроцессинг и подготовка признаков (`test_preprocessing.py`);
- подбор и применение порога классификации (`test_threshold.py`);
- сценарии CLI-инференса (`test_predict_cli.py`);
- корректность работы API-эндпоинтов (`test_api.py`);
- мониторинг дрифта признаков (`test_drift.py`);
- настройка логирования и вспомогательные функции (`test_logger_setup.py`, `test_utils.py`).

Запуск тестов выполняется автоматически в CI-пайплайне, что снижает риск регрессий при изменениях кода.

### Воспроизводимый запуск

Для воспроизводимости среды выполнения и изолированного запуска сервиса используется Docker.

Docker-образ содержит необходимые зависимости и позволяет запускать сервис инференса
в одинаковом окружении, что упрощает проверку и демонстрацию проекта.

## Мониторинг дрифта признаков (PSI)

В проекте реализован отдельный модуль offline-мониторинга дрифта признаков,
предназначенный для анализа стабильности входных данных после обучения модели.

Мониторинг используется для оценки изменений распределений признаков в новых данных 
по сравнению с обучающей выборкой и реализован как независимый batch-сценарий.

В качестве метрики дрифта используется **Population Stability Index (PSI)**.
Сравнение выполняется между обучающей выборкой (reference dataset) и новой партией данных (current dataset).

Расчёт дрифта производится на уровне исходных признаков
(до One-Hot Encoding), что упрощает интерпретацию результатов.

Для числовых признаков PSI рассчитывается по квантильным бинам, сформированным на reference-данных,
с учётом отдельного бина для пропусков.
Для категориальных признаков используются доли категорий, включая новые (ранее не наблюдавшиеся) значения.

Для каждого признака формируется агрегированный отчёт,
содержащий значение PSI, статус дрифта (`OK / WARN / DRIFT`), 
а также вспомогательные статистики по пропускам и количеству уникальных значений.

Модуль предназначен для аналитического контроля данных и не является частью онлайн-инференса.

**Запуск мониторинга**

```
python -m src.monitoring.drift
```

После выполнения формируется **CSV-отчёт**:

```
reports/drift_report.csv
```

## Данные

В проекте используются данные из соревнования Kaggle,
которые не хранятся в репозитории и загружаются локально через Kaggle API.

Для работы с проектом необходимо:
1) получить репозиторий с кодом;
2) настроить доступ к данным Kaggle;
3) загрузить обучающую и тестовую выборки в локальную директорию проекта.

Ниже описана полная последовательность действий.

### 1) Получение проекта

Склонируйте репозиторий и перейдите в его корневую директорию:

```
git clone https://github.com/lid11a/loan-payback-predictor.git
cd loan-payback-predictor
```

После выполнения команды `cd` вы находитесь в корне проекта —
в директории, содержащей `src/`, `tests/`, `notebooks/`, `Dockerfile` и `README`.

### 2) Настройка доступа к данным Kaggle

Для загрузки данных используется **Kaggle CLI** и персональный API-токен.

  1. Перейдите на сайт Kaggle и откройте настройки аккаунта: **Profile → Account**.

  2. В разделе **API** нажмите **Create New API Token**. 
  Будет скачан файл `kaggle.json`, содержащий API-ключи. 

  3. Поместите файл `kaggle.json` в стандартную директорию Kaggle:

    - **Windows:** `%USERPROFILE%\.kaggle\kaggle.json`
    - **Linux / macOS:** `~/.kaggle/kaggle.json`

  Для Linux/macOS также необходимо установить права доступа:

  ```
  chmod 600 ~/.kaggle/kaggle.json
  ```

### 3) Загрузка и расположение данных

После настройки Kaggle API данные загружаются автоматически
при запуске обучения модели либо могут быть загружены заранее вручную.

По умолчанию после загрузки в проекте создаются файлы:

- `data/raw/train.csv` — обучающая выборка;
- `data/raw/test.csv` — тестовая выборка.

Если файлы уже присутствуют локально, повторная загрузка не выполняется.

## Запуск проекта

В этом разделе описан стандартный порядок работы с проектом — 
от установки зависимостей до обучения модели и выполнения инференса. 
Все команды выполняются из корневой директории репозитория. 
Перед началом работы необходимо настроить доступ к Kaggle API и загрузить данные 
в соответствии с инструкциями из раздела **Данные**.



### 1) Установка зависимостей

Проект использует Python и виртуальное окружение.
Рекомендуется создать отдельное окружение перед установкой зависимостей.

**Windows (PowerShell)**

```
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements-all.txt
```

**macOS / Linux**

```
python -m venv .venv
source .venv/bin/activate
pip install -r requirements-all.txt
```

### 2) Запуск тестов

Перед запуском обучения рекомендуется проверить корректность установки и базовую работоспособность проекта.

```
pytest
```

### 3) Обучение модели

Обучение финальной модели запускается следующей командой:

```
python -m src.models.train
```

В процессе обучения:

- при отсутствии локальных данных выполняется их автоматическая загрузка из Kaggle;
- формируется единый пайплайн препроцессинга данных;
- обучается бинарный классификатор на основе LightGBM;
- подбирается порог классификации в соответствии с заданным бизнес-ограничением
  (целевой FPR, по умолчанию `target_fpr = 0.20`);
- сохраняется итоговый bundle модели в `models/best_model.joblib`
  (модель, препроцессор, порог классификации и метаданные).

### 4) Оффлайн-инференс

После обучения модели можно выполнить инференс на тестовых данных:

```
python -m src.models.predict
```

Результаты сохраняются в директории `data/predictions/` и включают:

- файл с вероятностями предсказаний (`data/predictions/submission_model.csv`);
- файл с вероятностями и бинарным решением по выбранному порогу (`data/predictions/decisions_model_thr_XXXX.csv`).

### 5) Запуск API локально (без Docker)

Для онлайн-инференса доступен HTTP API на базе FastAPI:

```
uvicorn src.api.app:app --host 0.0.0.0 --port 8000
```

**Swagger UI** доступен по адресу:

http://127.0.0.1:8000/docs

**Проверка** готовности сервиса (загружена ли модель):

http://127.0.0.1:8000/ready

### 6) API эндпоинты

HTTP API предназначен для онлайн-инференса модели и реализован на базе FastAPI.

Доступные эндпоинты:

- `GET /health` — проверка доступности сервиса;
- `GET /ready` — проверка готовности модели к инференсу;
- `GET /features` — список ожидаемых входных признаков;
- `POST /predict` — инференс для одной записи;
- `POST /predict_batch` — инференс для набора записей.

Интерактивная документация доступна через **Swagger UI**:

http://127.0.0.1:8000/docs

#### Пример запроса к API

Перед отправкой запроса посмотрите, какие признаки ожидает модель (`GET /features`),
и какие значения допустимы для категориальных признаков (`GET /features/schema`).

Пример запроса для онлайн-инференса одной записи:

```
POST /predict
{
  "features": {
    "gender": "Female",
    "marital_status": "Single",
    "education_level": "Bachelor",
    "employment_status": "Employed",
    "loan_purpose": "Debt Consolidation",
    "grade_subgrade": "B2",
    "annual_income": 50000,
    "debt_to_income_ratio": 0.25,
    "credit_score": 680,
    "loan_amount": 12000,
    "interest_rate": 12.5
  }
}
```

### 7) MLflow — управление экспериментами

В процессе обучения используется MLflow для отслеживания и воспроизводимости экспериментов.

В рамках каждого обучающего запуска в MLflow фиксируются:

- параметры модели и обучения;
- метрики качества;
- артефакты, связанные с обученной моделью.

Для просмотра истории экспериментов и сравнения запусков
можно воспользоваться web-интерфейсом MLflow:

```
mlflow ui
```

По умолчанию интерфейс будет доступен по адресу:

http://127.0.0.1:5000

### 8) Docker — воспроизводимый запуск сервиса

Для воспроизводимого и изолированного запуска сервиса предусмотрено Docker-окружение.

#### Сборка Docker-образа

Для сборки Docker-образа требуется доступ к Docker Hub
(для загрузки базового образа python:3.12-slim):

```
docker build -t loan-api .
```
В ограниченных сетевых окружениях (например, в корпоративных сетях или при ограничениях со стороны интернет-провайдера)
сборка Docker-образа может быть недоступна из-за отсутствия доступа к Docker Hub.
В этом случае API можно запускать локально без использования Docker.

#### Запуск контейнера

Папка `models/` не хранится в репозитории, поэтому её нужно примонтировать в контейнер, 
чтобы сервис имел доступ к сохранённой модели.

**Windows (PowerShell)**

Выполните две команды:

```
$models = (Resolve-Path .\models).Path
docker run --rm -p 8000:8000 -v "${models}:/app/models" loan-api
```

**macOS / Linux**

```
docker run --rm -p 8000:8000 -v "$(pwd)/models:/app/models" loan-api
```

После запуска API и Swagger UI будут доступны по тем же адресам,
что и при локальном запуске без Docker:

http://127.0.0.1:8000/docs

### 9) CI (GitHub Actions)

В репозитории настроен CI-пайплайн на базе **GitHub Actions**.

CI автоматически запускается при **push** и **pull request** и выполняет:
- установку зависимостей (из `requirements-ci.txt`);
- запуск тестов (`pytest`);
- расчёт покрытия тестами (coverage).

Статус пайплайна отображается в бейдже в начале README.

## Используемые технологии

В проекте применён набор инструментов, охватывающий полный цикл работы с ML-задачей: 
от исследовательского анализа данных до сервисного инференса и автоматизированной проверки кода.

### ML-пайплайн и сервисная часть (`src/`)

- **Язык и базовые библиотеки**  
  Python, pandas, numpy — работа с данными и численными вычислениями.

- **Машинное обучение и препроцессинг**  
  scikit-learn — построение пайплайнов обработки данных, кодирование признаков, разбиение выборки, метрики качества.

- **Финальная модель**  
  LightGBM — градиентный бустинг для задачи бинарной классификации;  
  обучение с использованием кросс-валидации и early stopping,
  работа с дисбалансом классов.

- **Сохранение модели и артефактов**  
  joblib — сохранение единого model bundle
  (модель, препроцессор, порог классификации, метаданные).

- **Логирование и управление экспериментами**  
  logging — логирование выполнения пайплайна и сервисной логики;  
  MLflow — трекинг параметров, метрик и артефактов обучающих запусков.

- **Тестирование и контроль качества**  
  pytest, coverage / pytest-cov — автоматическое тестирование ключевой логики и контроль покрытия тестами.

- **Получение данных**  
  Kaggle CLI — автоматизированная загрузка исходных данных.

- **Сервис для инференса**  
  FastAPI, Uvicorn — HTTP API для онлайн-предсказаний.

- **Воспроизводимость и автоматизация**  
  Docker — изолированный и воспроизводимый запуск сервиса;  
  GitHub Actions — CI-пайплайн для автоматической проверки кода при изменениях.

### Исследовательская часть и эксперименты (`notebooks/`)

В исследовательской части проекта использовались различные методы анализа данных
и классы алгоритмов для изучения структуры данных, проверки гипотез и выбора финального решения.

- **Анализ данных и статистика**  
  pandas, numpy, scipy — анализ распределений признаков, пропусков и целевой переменной, 
  проверка статистических зависимостей (в том числе chi-square тесты).

- **Визуализация и EDA**  
  matplotlib, seaborn, plotly — визуальный анализ распределений, сравнение сегментов и классов.

- **Кластеризация (исследовательский анализ)**  
  KMeans, Gaussian Mixture Models (GMM) — изучение структуры данных, подбор числа кластеров по silhouette score,
  PCA-визуализация и анализ распределения таргета внутри кластеров.

- **Линейные модели**  
  Logistic Regression.

- **Методы, основанные на расстояниях и разделяющих поверхностях**  
  K-Nearest Neighbors, Support Vector Machines.

- **Вероятностные модели**  
  Naive Bayes.

- **Деревья решений и ансамблевые методы**  
  Random Forest.

- **Градиентный бустинг**  
  XGBoost, CatBoost, LightGBM.

- **Нейронные сети**  
  Полносвязная нейронная сеть (MLP).

- **Инфраструктура экспериментов**  
  Jupyter Notebook.
